## 1. 에셋 번들 시스템 개선

라이브 운영 중 팝업 로딩 지연, 리소스 충돌, 번들 의존성 관리 어려움 등의 문제가 반복적으로 발생했습니다.
단기적인 지연 제거보다, **번들 로딩 흐름과 참조 관계 자체를 정리하는 것**을 목표로 구조를 재설계했습니다.

### 문제 인식
* 팝업 프리팹이 빌트인 리소스로 포함되어 빌드 용량 증가
* 팝업 표시 시 리소스 로드 → 빌트인 프리팹 바인딩 구조로 평균 0.5초 이상 지연
* 번들 간 의존성이 명확하지 않아 중복 리소스 및 로드 충돌 발생

### 개선 내용
* 팝업 호출 구조 전면 리팩토링
* 팝업 프리팹을 에셋 번들로 분리하고 메모리 캐싱 기반 재사용 구조로 전환
* 번들 포맷을 LZ4 기반으로 변경하여 로딩 속도 개선
* 공통 리소스를 `common_resource` 번들로 분리하여 의존성 단순화

### 결과
* 팝업 로딩 지연 현상 제거 (0.5s → 즉시 출력 수준)
* 중복 에셋 제거를 통해 전체 에셋 번들 용량 절감
* 리소스 참조 오류로 인한 런타임 예외 발생 상황 차단

## 2. 맵 데이터 저장 구조 개선 및 배포 자동화 (Jenkins)

대규모 맵 데이터를 다루는 과정에서 수동 작업으로 인한 병목과 휴먼 에러를 해결하기 위해 파이프라인을 재구축했습니다.

### 문제 인식
* 기존에는 작업자가 로컬 PC에서 수동으로 맵 데이터를 취합하고 검증하여 업로드하는 방식
* 데이터 확인 및 배포 프로세스에 많은 시간이 소요되며, 작업자 개인의 환경에 의존적임

### 개선 내용
* **Jenkins + Slack 기반 CI/CD 구축**: 버튼 클릭 한 번으로 데이터 빌드 및 서버 배포가 완료되는 환경 조성
* 데이터 배포 전 포맷 오류 및 유효성을 자동으로 체크하는 검증 단계 추가

### 결과
* **운영 효율 향상**: 기획자 업무 기준, 기존 **3일이 소요되던 데이터 작업 프로세스를 1일로 단축 (66% 개선)**
* 휴먼 에러로 인한 빌드 오류 발생 및 라이브 사고 리스크 원천 차단

---

## 3. JSON 시스템 개선

기존 시스템은 JSON 데이터를 단순 문자열 파싱으로 처리하여, 중첩 구조 처리와 유지보수 측면에서 한계가 있었습니다. 이를 구조화된 모델 기반 파싱 방식으로 전환했습니다.

### 개선 내용
* Unity.Serialization.Json 기반 파싱 구조로 전환
* JSON → 객체 직렬화 구조 도입
* 반복 파싱 로직 제거 및 모델 단위 관리

### 결과
* 파싱 속도 개선 (3000회 기준 1200ms → 985ms)
* 메모리 사용량 감소 및 데이터 구조 변경 시 대응 비용 감소

## 4. 월드맵 드로우콜 최적화

월드맵 UI 요소 증가로 드로우콜이 급격히 늘어나는 문제가 있었습니다.

### 문제 인식

* 팝업 및 이벤트 UI 추가 시 드로우콜 기하급수적 증가
* 공통 리소스가 프리팹별로 중복 포함

### 개선 내용

* 공통 UI 요소 Canvas 분리 및 풀링 구조 도입
* 공통 스프라이트 Atlas 통합
* 기본 상태 드로우콜 수 기준 설정 및 유지

### 결과

* 기존 batch수 170초과에서 110 ~100 유지
* 메모리 사용량 및 프레임 저하 방지

## 6. Burst Compiler 기반 고성능 연산 (회전팡 최적 각도 계산)

인게임 특수 블록 사용 시, 가장 효율적인 타격 각도를 실시간으로 계산하기 위해 고성능 연산 구조를 도입했습니다.

### 문제 인식
* 81개 셀에 대해 모든 각도의 시뮬레이션을 수행해야 하므로, 일반적인 루프 연산 시 모바일 기기에서 프레임 드롭 발생

### 개선 내용
* **비트마스크 최적화**: 81개 셀의 상태를 직접 구현한 `UInt128` 구조체 기반의 비트 데이터로 관리하여 연산량 최소화
* **Burst & Job System**: 연산 로직을 네이티브 코드로 컴파일하고 멀티 스레드로 분산 처리

### 결과
* 복잡한 전 방향 시뮬레이션을 매 프레임 수행함에도 프레임을 안정적으로 유지
* 오토플레이 시스템 및 사용자 편의 기능의 기술적 기반 마련

---

## 7. 셔플 시 특수 블록 조합 배치

셔플 발생 시 단순 랜덤이 아닌, 한 번의 스왑으로 5개 일렬 매칭(회전팡)을 만들 수 있는 패턴이 배치되도록 개선했습니다.

* 가로/세로 기준 12가지 프리-패턴을 도출하여 셔플 시 보드에 배치 가능 여부를 검증 후 적용
* 패턴 배치 불가 시 일반 셔플로 폴백 처리

## 회고

최적화 작업을 진행하며 느낀 점은,
성능 문제는 단일 코드 수정으로 해결되는 경우보다
**구조와 관리 방식이 누적되며 드러나는 경우가 훨씬 많다**는 점이었습니다.

이후에는 성능 개선을 개별 기술 적용이 아니라,
"운영 중 비용이 반복적으로 발생하는 지점을 구조적으로 제거하는 작업"으로 인식하게 되었고,
이 관점은 이후 시스템 설계와 도구 개발 전반에 영향을 주고 있습니다.
