# 시스템 최적화 및 자동화 경험 정리

본 문서는 라이브 서비스 중 실제로 수행한 **성능 최적화 및 운영 효율 개선 작업**을 정리한 자료입니다.
기술 나열보다는, 어떤 문제를 인지했고 어떤 판단으로 구조를 정리했는지를 중심으로 서술했습니다.

---

## 작업 범위 요약

* 에셋 번들 로딩 구조 및 의존성 정리
* 대규모 맵 데이터 저장·배포 구조 개선
* JSON 파싱 구조 개편
* 월드맵 UI 드로우콜 최적화
* 에디터 자동화 및 반복 작업 제거
* 오토플레이 및 퍼즐 로직 보조 시스템

---

## 1. 에셋 번들 시스템 개선

라이브 운영 중 팝업 로딩 지연, 리소스 충돌, 번들 의존성 관리 어려움 등의 문제가 반복적으로 발생했습니다.
단기적인 지연 제거보다, **번들 로딩 흐름과 참조 관계 자체를 정리하는 것**을 목표로 구조를 재설계했습니다.

### 문제 인식

* 팝업 프리팹이 빌트인 리소스로 포함되어 빌드 용량 증가
* 팝업 표시 시 리소스 로드 → 빌트인 프리팹 바인딩 구조로 평균 0.5초 이상 지연
* 번들 간 의존성이 명확하지 않아 중복 리소스 및 로드 충돌 발생

### 개선 내용

* 팝업 호출 구조 전면 리팩토링
* 팝업 프리팹을 에셋 번들로 분리하고 메모리 캐싱 기반 재사용 구조로 전환
* 번들 포맷을 LZ4 기반으로 변경하여 로딩 속도 개선
* 공통 리소스를 `common_resource` 번들로 분리하고 단방향 의존성 규칙 도입

### 결과

* 팝업 표시 시 체감 지연 제거
* 번들 의존 관계 추적 용이
* 리소스 중복 감소 및 유지보수 부담 완화

---

## 2. 맵 데이터 저장 구조 개선

5000개 이상의 맵 데이터를 안정적으로 관리하기 위해,
기존 NAS 기반 수동 관리 방식에서 **Git 기반 버전 관리 구조**로 전환했습니다.

### 문제 인식

* NAS 수동 병합으로 인한 작업 중단 및 충돌 빈번
* 변경 이력 추적 및 롤백 어려움
* JSON 기반 대용량 데이터로 인한 로딩 부담

### 개선 내용

* 맵 데이터를 스테이지 단위 파일로 분리하여 Git으로 관리
* 기획자 대상 Git 사용 방식 공유 및 구조 전환 리딩
* LZ4 압축 기반 바이너리 포맷 도입
* Jenkins + Slack 연동을 통한 맵 데이터 취합 및 배포 자동화
* 기존 통합 JSON 맵 데이터를 스테이지 단위로 분해하여 마이그레이션

### 결과

* 맵 데이터 변경 이력 추적 가능
* 데이터 취합 및 배포 시간 단축
* 로딩 성능 및 데이터 안정성 개선

---

## 3. JSON 시스템 개선

기존 SimpleJson 기반 파싱 구조는 중첩 구조 처리와 유지보수 측면에서 한계가 있었습니다.
이를 구조화된 모델 기반 파싱 방식으로 전환했습니다.

### 개선 내용

* Unity.Serialization.Json 기반 파싱 구조로 전환
* JSON → 객체 직렬화 구조 도입
* 반복 파싱 로직 제거 및 모델 단위 관리

### 결과

* 파싱 속도 개선 (3000회 기준 1200ms → 985ms)
* 메모리 사용량 감소
* 데이터 구조 변경 시 대응 비용 감소

---

## 4. 월드맵 드로우콜 최적화

월드맵 UI 요소 증가로 드로우콜이 급격히 늘어나는 문제가 있었습니다.

### 문제 인식

* 팝업 및 이벤트 UI 추가 시 드로우콜 기하급수적 증가
* 공통 리소스가 프리팹별로 중복 포함

### 개선 내용

* 공통 UI 요소 Canvas 분리 및 풀링 구조 도입
* 공통 스프라이트 Atlas 통합
* 기본 상태 드로우콜 수 기준 설정 및 유지

### 결과

* 팝업 수와 무관하게 드로우콜 100 이하 유지
* 메모리 사용량 및 프레임 저하 방지

## 5. 에디터 자동화 및 커스텀 컴포넌트

아트 직군과의 협업 과정에서 발생하는 반복 작업을 줄이기 위해,
에디터 기반 자동화 툴과 공용 컴포넌트를 개발했습니다.

### 개선 내용

* 폴더 업로드 기반 자동 번들 제작 에디터 툴 구현
* Atlas 생성 및 리소스 가공 자동화
* 반복 사용되는 UI 구조를 커스텀 Inspector 컴포넌트로 모듈화

### 결과

* 리소스 처리 과정 단순화
* 팀 내 반복 작업 감소 및 생산성 향상

## 회고

최적화 작업을 진행하며 느낀 점은,
성능 문제는 단일 코드 수정으로 해결되는 경우보다
**구조와 관리 방식이 누적되며 드러나는 경우가 훨씬 많다**는 점이었습니다.

이후에는 성능 개선을 개별 기술 적용이 아니라,
"운영 중 비용이 반복적으로 발생하는 지점을 구조적으로 제거하는 작업"으로 인식하게 되었고,
이 관점은 이후 시스템 설계와 도구 개발 전반에 영향을 주고 있습니다.
